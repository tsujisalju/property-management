## What this project is

A cloud-based property management platform for the DDAC university assignment.
It addresses three real-world problems in the property management industry:

- **Tenant communication** — tenants can submit and track issues directly in the platform
- **Maintenance and repairs** — managers schedule work, assign staff, and track history
- **Financial management** — transparent budgets, invoices, and payment tracking

The project is assessed across two tasks:
- **Task 1** (current focus): frontend + backend + RDS database, deployed to EC2
- **Task 2** (later): add serverless components (Lambda, API Gateway, S3/SNS/SQS) and CloudWatch monitoring

## Tech stack

| Layer | Technology | Where it runs |
|---|---|---|
| Frontend | Next.js 15, TypeScript, React 19, Tailwind CSS | Vercel (prod) / localhost:3000 (dev) |
| Backend | ASP.NET 8, C# | AWS EC2 in Docker (prod) / localhost:8080 (dev) |
| Database | PostgreSQL 15 via EF Core | AWS RDS (prod) / Docker Compose (dev) |
| File storage | AWS S3 | Pre-signed URL pattern |
| Email | AWS SES v2 | Via `IEmailService` |
| Auth | AWS Cognito (JWT) | Planned — stubs in place |
| Monitoring | AWS CloudWatch + X-Ray | Task 2 |

**Hard constraint:** the frontend must never call AWS services directly. All AWS
operations (S3, SES, RDS) go through the ASP.NET backend. The frontend only calls
`/api/*` endpoints on the backend.

## Running the project locally

```bash
cp .env.example .env   # fill in AWS credentials
docker compose up --build
```

- Frontend: http://localhost:3000
- Backend API + Swagger UI: http://localhost:8080/swagger
- Health check: `curl http://localhost:8080/api/health`

The Postgres database is auto-migrated on backend startup and seeded with four
dev users (one per role) from `db/migrations/001_initial_schema.sql`.

To wipe and reseed the database:
```bash
docker compose down -v && docker compose up --build
```

## Database schema (summary)

All tables use UUID primary keys generated by `gen_random_uuid()`.

| Table | Purpose | Key relationships |
|---|---|---|
| `users` | All user accounts across all roles | Has `cognito_sub` for auth linkage |
| `properties` | A managed building or complex | `manager_id → users` |
| `units` | Individual rentable unit within a property | `property_id → properties` |
| `leases` | Tenancy record linking a tenant to a unit | `unit_id → units`, `tenant_id → users` |
| `maintenance_requests` | Tenant-submitted issue or scheduled work | `unit_id`, `tenant_id`, `assigned_to → users` |
| `maintenance_comments` | Thread of messages on a request | `request_id`, `author_id → users` |
| `invoices` | Rent or maintenance billing record | `lease_id → leases` |
| `budgets` | Monthly budget allocation per category | `property_id → properties` |

`MaintenanceRequest` has **two FKs to `users`** (`TenantId` and `AssignedTo`).
These are explicitly configured in `AppDbContext.OnModelCreating` — do not remove
those configurations or EF Core will throw on startup.

String enum values in use:
- `users.role`: `manager | tenant | maintenance_staff | admin`
- `units.status`: `vacant | occupied | maintenance`
- `leases.status`: `active | expired | terminated`
- `maintenance_requests.status`: `open | in_progress | resolved | closed`
- `maintenance_requests.priority`: `low | medium | high | emergency`
- `maintenance_requests.category`: `plumbing | electrical | hvac | general`
- `invoices.status`: `pending | paid | overdue | cancelled`
- `invoices.type`: `rent | maintenance | deposit | penalty`

## Backend conventions

### Adding a new controller

1. Create `Controllers/XyzController.cs` inheriting `ControllerBase`
2. Annotate with `[ApiController]` and `[Route("api/[controller]")]`
3. Inject `AppDbContext` and any services via the primary constructor
4. Use `async Task<IActionResult>` for all action methods
5. Return DTOs, never entity classes directly
6. Add corresponding DTO records to `DTOs/Dtos.cs`

```csharp
[ApiController]
[Route("api/properties")]
public class PropertiesController(AppDbContext db) : ControllerBase
{
    [HttpGet]
    public async Task<IActionResult> GetAll() { ... }
}
```

### Adding a new service

1. Define an interface `IXyzService` and implementation `XyzService` in `Services/`
2. Register in `Program.cs`: `builder.Services.AddScoped<IXyzService, XyzService>()`
3. Inject via constructor in controllers that need it

### S3 file references

Files are never stored in the database. The pattern is:
1. Backend generates a pre-signed upload URL via `IS3Service.GetUploadUrlAsync(key, contentType)`
2. Frontend uploads directly to S3 using that URL (PUT request, no auth header)
3. Frontend calls PATCH on the backend to save the S3 key to the database record
4. To serve a file, backend generates a pre-signed download URL via `IS3Service.GetDownloadUrlAsync(key)`

S3 keys follow the convention: `{domain}/{record-id}/{filename}`, e.g.
`maintenance/abc-123/photo-xyz.jpg` or `invoices/abc-123/invoice.pdf`.

### Auth (currently stubbed)

Cognito JWT auth is wired in `Program.cs` but not enforced on any route yet.
Controllers have `// TODO: replace with authenticated user ID from JWT` comments
where the current user's ID needs to be read. When implementing auth:
- Read `User.FindFirst("sub")?.Value` to get the Cognito sub
- Look up the `users` row by `cognito_sub` to get the internal `Guid` user ID
- Gate list endpoints by role — tenants should only see their own records

## Frontend conventions

### Making API calls

Always go through `lib/api.ts` — never write raw `fetch` calls in page or component
files. If an endpoint doesn't exist in `api.ts` yet, add it there first.

```typescript
// In a Server Component (default in App Router):
import { maintenanceApi } from "@/lib/api";
const requests = await maintenanceApi.list({ status: "open" });

// In a Client Component ("use client"):
import { maintenanceApi } from "@/lib/api";
const requests = await maintenanceApi.list();
```

### Adding a new page

Create the file at `app/{route}/page.tsx`. Prefer Server Components (no
`"use client"`) for data fetching — they fetch on the server at request time
and need no loading state. Use Client Components only for interactivity
(forms, dropdowns, real-time updates).

### TypeScript types

All types live in `types/index.ts` and mirror the backend DTOs exactly.
When you add a new DTO to the backend, add the matching TypeScript type here.

### Styling

Use Tailwind utility classes. Do not write custom CSS except in `globals.css`
for truly global resets. Colour palette in use: `indigo` for primary actions,
`gray` for neutral surfaces, semantic colours (`red`, `green`, `amber`) for
status indicators only.

## Environment variables

| Variable | Used by | Purpose |
|---|---|---|
| `AWS_ACCESS_KEY_ID` | Backend | AWS auth (dev only — prod uses IAM role) |
| `AWS_SECRET_ACCESS_KEY` | Backend | AWS auth (dev only) |
| `AWS_REGION` | Backend | Default: `ap-southeast-1` |
| `AWS_S3_BUCKET` | Backend (`S3Service`) | Bucket for all file uploads |
| `AWS_SES_SENDER` | Backend (`EmailService`) | Verified sender address |
| `COGNITO_USER_POOL_ID` | Backend | Cognito pool for JWT validation |
| `COGNITO_CLIENT_ID` | Backend | Cognito app client |
| `NEXT_PUBLIC_API_URL` | Frontend | Backend base URL (empty string in dev — rewrites handle it) |

Never hardcode credentials. Never commit `.env`.

## What is already built

- `GET /api/maintenance-requests` — list with `?status=` and `?unitId=` filters
- `GET /api/maintenance-requests/{id}` — single request with comments

- `PATCH /api/maintenance-requests/{id}` — update status, assignee, priority

- `IEmailService` / `EmailService` — send HTML email via SES
- EF Core models for all 8 database tables
- All DTOs for all planned endpoints
- Frontend API client stubs for all domains
